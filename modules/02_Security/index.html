Access Control

  • Can entity perform action on object?

  • Authentication:

    • Verify credentials of entity.

    • Ensure the identity of the entity is valid.

  • Authorization:

    • What actions an entity can perform.

Authentication

  • Transport authentication:

    • Service-to-service authentication.

    • Verifies the direct client making the conmnection.

    • Example: mTLS

  • Origin Authentication

    • End-user authentication

    • Verifies the original client (end-user, device) making the request.

    • OpenID Connect with JSON Web Token (JWT).

Goals of Istio Security

  • Security by default: no changes needed for application code and infrastructure.

  • Defense in depth: integrate with existing security systems to provide multiple layers of defense.

  • Zero-trust network: build security solutions on untrusted networks.

Istio Security Architecture

Identity

  • Fundamental aspect of any security infrastructure.

  • Secure service-to-service authentication:

    • Services exchange credentials with identity information.

    • Client side: checks server identity, verifies if it is authorized runner of the service.

    • Server side: determine what the client is authorized to do according to authorization policies.

Istio Identity Model

  • Based on SPIFFE standard

    • "Secure Production Identity Framework for Everyone"

    • Set of open-source standards for securely identifying software systems in dynamic and heterogeneous environments.

  • Istio Identity:

    • Based on Kubernetes Service Account.

    • spiffe://<domain>/ns/<namespace>/sa/<serviceaccount>

 

Istio PKI

  • Built on top of Istio Citadel

  • Securely provisions strong identities to every workload.

  • Uses X.509 certificates to carry the identities in SPIFFE format.

  • Automates the key & certificate rotation.

 

Istio PKI: Certificate Secret

  • Citadel watches the Kubernetes apiserver.

  • Creates a SPIFFE certificate and key pair for each of the existing and new service accounts.

  • Stores the certificate and key pairs as Kubernetes secrets.

  • When a pod is created, certificate secret is mounted into the pod according to its service account.

  • Citadel watches the lifetime of the certificate, and automatically rotates the certificates by rewriting the Kubernetes secrets and restarting the Envoy proxy.

  • Pilot generates the secure naming information

    • Defines what service account or accounts can run a certain service.

Istio PKI: Node Agent + SDS

  • SDS: Secret Discovery Service.

    • Remote protocol for Envoy proxy to fetch certificates.

  • Node Agent: gRPC service which takes CSR requests. Runs as DaemonSet on each node.

  • When a pod is created, Envoy sends a certificate and key request via Envoy SDS API to the Node Agent.

  • The node agent creates the private key and CSR and sends the CSR with its credentials to Citadel for signing.

  • Citadel validates the credentials carried in the CSR and signs the CSR to generate the certificate.

  • The node agent sends the certificate received from Citadel and the private key to Envoy via the Envoy SDS API.

  • Process repeats periodically for certifcate and key rotation.

Istio PKI: Node Agent + SDS

  • Advantages of SDS:

    • The private key never leaves the node: It is only in the Citadel agent and Envoy sidecar’s memory.

    • The secret volume mount is no longer needed: The reliance on the Kubernetes secrets is eliminated.

    • The sidecar Envoy is able to dynamically renew the key and certificate through the SDS API: Certificate rotations no longer require Envoy to restart.

Istio PKI: Node Agent + SDS

  • Not supported on the current version of Red Hat Service Mesh.

  • Reason: security risks associated with use of Unix Domain Socket.

  • Current upstream implementation uses Unix Domain Socket for connection between Node Agent and the Envoy Proxy.

  • All pods running in the same Kubernetes node share the Citadel agent and Unix domain socket.

  • A malicious pod could hijack the Unix domain socket to break the SDS service or steal the identity credentials from other pods running on the same Kubernetes node.

  • Might be supported on next release.

Istio mTLS

  • mTLS: provide two-way authentication and encryption.

  • With Istio services can engage in mTLS communication without requiring to implement mTLS at the application level.

  • For a client to call a server with mutual TLS authentication:

    • Istio re-routes the outbound traffic from a client to the client’s local sidecar Envoy.

    • The client side Envoy starts a mutual TLS handshake with the server side Envoy.

    • During the handshake, the client side Envoy also does a secure naming check to verify that the service account presented in the server certificate is authorized to run the target service.

    • The client side Envoy and the server side Envoy establish a mutual TLS connection, and Istio forwards the traffic from the client side Envoy to the server side Envoy.

    • After authorization, the server side Envoy forwards the traffic to the server service through local TCP connections.

Authentication Policies

  • CRD: Policy, ServiceMeshPolicy.

  • Example:
apiVersion: "authentication.istio.io/v1alpha1"
kind: "Policy"
metadata:
  name: "reviews"
spec:
  targets:
  - name: reviews
  peers:
  - mtls: {}

  • Policy scope: service-specific > namespace-wide > mesh-wide

  • mTLS values: STRICT, PERMISSIVE

Origin Authentication

  • origins section in Policy resource.

  • Istio only supports JWT authentication

  • JWT token is passed to the application by the service proxy

  • JWT token propagation is the responsibility of the application

  • Example:
origins:
- jwt:
    issuer: "https://accounts.google.com"
    jwksUri: "https://www.googleapis.com/oauth2/v3/certs"
    trigger_rules:
    - excluded_paths:
      - exact: /health

Authorization

  • Enabled with ServiceMeshRbacConfig resource.

  • Example:

apiVersion: rbac.maistra.io/v1
kind: SeviceMeshRbacConfig
metadata:
  name: default
spec:
  mode: 'ON_WITH_INCLUSION'
  inclusion:
    namespaces: ["default"]

  • mode values:

    • OFF: authorization is disabled.

    • ON: authorization is enabled for all services in the mesh.

    • ON_WITH_INCLUSION: authorization is enabled only for services and namespaces specified in the inclusion field.

    • ON_WITH_EXCLUSION: authorization is enabled for all services in the mesh except the services and namespaces specified in the exclusion field.

Authorization Policy

  • Defined by ServiceRole and ServiceRoleBinding resources.

  • ServiceRole: defines a group of permissions to access services.

  • ServiceRoleBinding: grants a ServiceRole to particular subjects, such as a user, a group, or a service.

  • who is allowed to do what under which conditions.

  • Next release of Service Mesh will be based on Istio 1.4, which supports AuthorizationPolicy CR.

ServiceRole

  • Example:

apiVersion: "rbac.istio.io/v1alpha1"
kind: ServiceRole
metadata:
  name: products-viewer
  namespace: ns1
spec:
  rules:
  - services: ["products.ns1.svc.cluster.local"]
    methods: ["GET", "HEAD"]

ServiceRoleBinding

  • Includes two parts:

    • roleRef refers to a ServiceRole resource in the same namespace.

    • A list of subjects that are assigned to the role.

  • ​Example:

apiVersion: "rbac.istio.io/v1alpha1"
kind: ServiceRoleBinding
metadata:
  name: test-binding-products
  namespace: ns1
spec:
  subjects:
  - user: "cluster.local/ns/istio-system/sa/istio-ingressgateway-service-account
  roleRef:
    kind: ServiceRole
    name: "products-viewer"

Authorization On Plain TCP Protocols

  • Authorization also supported for non-HTTP traffic

  • ​Example:

apiVersion: "rbac.istio.io/v1alpha1"
kind: ServiceRole
metadata:
  name: mongodb-viewer
  namespace: ns1
spec:
  rules:
  - services: ["mongodb.ns1.svc.cluster.local"]
    constraints:
    - key: "destination.port"
      values: ["27017"]
---
apiVersion: "rbac.istio.io/v1alpha1"
kind: ServiceRoleBinding
metadata:
  name: bind-mongodb-viewer
  namespace: ns1
spec:
  subjects:
  - user: "cluster.local/ns/ns1/sa/bookinfo-ratings-v2"
  roleRef:
    kind: ServiceRole
    name: "mongodb-viewer"
