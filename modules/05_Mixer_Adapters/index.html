Mixer

  • Component responsible for providing policy controls and telemetry collection

    • check, report

  • Integrates Service Mesh with infrastructure backend systems

    • Telemetry capturing systems, access control systems, quota enforcement systems, billing systems ...

  • Uses plugin model and uniform abstraction to integrate with backend systems

 

Mixer Topology

Mixer Topology

  • Envoy proxy calls Mixer before each request to perform pre-condition checks.

    • Local cache to perform precondition checks locally

  • Envoy calls Mixer after each request to report telemetry.

    • Buffering of outgoing telemetry.

  • Policy enforcement and telemetry collection driven from configuration.

    • Instance, Handler, Rule resources

  • Policy check is disabled by default.

  • Policy check and telemetry collection handled by different deployments of Mixer.

    • istio-policy, istio_telemetry

 

Mixer Adapters

  • Mixer is a highly modular and extensible component.

  • Abstracts away the details of different policy and telemetry backend systems

  • Adapters allow Mixer to interface to different infrastructure backends

  • Comes with set of built-in plugins or adapters - e.g. Prometheus

  • Custom adapters can be added to mixer - e.g. 3Scale Mixer Adapter

    •  gRPC Adapters

    • Run as separate pods in control or management plane

  • Exact set of adapters used at runtime is determined through configuration

Attributes

  • Attribute = small bit of data that describes a single property of a specific service request.

    • Size, response code, source IP Address ...

  • Attribute has a name and a type

  • Service Mesh has fixed vocabulary of attributes it understands

  • Examples of attributes:

request.path: xyz/abc
request.size: 234
request.time: 12:34:56.789 04/17/2017
source.ip: [192 168 0 1]
destination.service.name: example

Attributes

Mixer is attribute processing machine.

  • Envoy invokes Mixer, sending set of attributes that describe the request and the request context.

  • Mixer generates calls to backend based on its configuration and given attributes.

Attribute Expressions

  • Mixer configuration uses expression language to specify match expressions and mapping expressions.

  • Expressions map a set of typed attributes and constants to a typed value.

  • Examples:

 

request.size | 200
request.headers["x-forwarded-host"] == "myhost"
(request.headers["x-user-group"] == "admin") || (request.auth.principal == "admin")
(request.auth.principal | "nobody" ) == "user1"
source.labels["app"]=="reviews" && source.labels["version"]

Mixer Configuration Model

  • Based on Adapters and Templates

    • Adapter: encapsulate the logic necessary to interface Mixer with a specific infrastructure backend.

    • Template: defines the schema for specifying request mapping from attributes to adapter inputs.

    • A given adapter may support any number of templates.

  • Configuration resources:

    • Handler: defines adapters being used and its configuration.

      • Example: define metric type for Prometheus

    • Instance: represents chunk of data an an adapter will operate on.

      • Example: generate request count metric from attributes destination.service.host and response.code

    • Rule: describe when an adapter is called and which instances it is given.

Mixer Configuration Example

apiVersion: config.istio.io/v1alpha2
kind: handler
metadata:
  name: version-count-handler
spec:
  compiledAdapter: prometheus
  params:
    metrics:
    - name: version_count
      instance_name: version-count.instance.er-demo
      kind: COUNTER
      label_names:
      - source
      - version
      - destination

Mixer Configuration Example

apiVersion: config.istio.io/v1alpha2
kind: instance
metadata:
  name: version-count
spec:
  compiledTemplate: metric
  params:
    value: "1"
    dimensions:
      source: source.workload.name | "unknown"
      version: destination.labels["version"] | "unknown"
      destination: destination.service.name | "unknown"

Mixer Configuration Example

apiVersion: config.istio.io/v1alpha2
kind: rule
metadata:
  name: version-count-prometheus
spec:
  match: (context.protocol == "http") && (destination.service.host == "incident-service.er-demo.svc.cluster.local")
  actions:
  - handler: version-count-handler
    instances:
    - version-count

3Scale Mixer Adapter

  • External gRPC adapter

  • Enabled and configured in ServiceMeshControlPlane CR

  • Enforce policies defined in 3Scale for traffic flowing through proxied services

  • Supports different authentication methods:

    • Standard API Keys

    • Application identifier and key pairs

    • OpenId: clientId parsed from JWT Token

  • ​Configurable caching of 3Scale Management API responses.

  • Exposes Prometheus metrics scraped by the Service Mesh Prometheus instance
